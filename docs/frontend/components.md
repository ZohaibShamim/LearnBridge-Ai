# Frontend — Shared Components

## What this is

These are reusable pieces of UI (user interface — the parts of the app you actually see and click) that get used on more than one page. Instead of rebuilding the navigation bar or the "skills you have vs. skills to gain" card from scratch on every screen, the app defines them once here and every page that needs them just imports and reuses them.

## File by file

### `components/landing/landing.tsx`

`LandingPage` — lines 29–170

This is the marketing/home page — the page a visitor sees before signing up. It's a single component with no props (inputs) at all; everything it shows is hardcoded inside the file, so there's no data fetching (no calling the backend) here.

It's built from a few stacked sections:
- **Navbar** (lines 33–54) — the "LearnBridge AI" logo/brand, links to the Learning Paths and Features sections on the same page, and "Sign In" / "Get Started" buttons that link to `/login` and `/signup`.
- **Hero** (lines 56–91) — the big banner at the top with a background image, the main headline ("Turn your resume into a personalized path forward"), and the two main call-to-action buttons.
- **Learning paths** (lines 93–113) — a grid of cards, one per career track (AI, Machine Learning, Data Science, Software Engineering). These cards are generated by looping (`.map()`) over a `paths` array defined at the top of the file (lines 14–19), so adding a new track just means adding one more entry to that array rather than writing new JSX (the HTML-like syntax React uses).
- **Features** (lines 115–137) — another grid, this time looping over a `features` array (lines 21–27), each with an icon, title, and short description (e.g. "Personalized Roadmaps", "Resume Analysis").
- **CTA (call to action)** (lines 139–154) — a colored banner nudging the visitor to sign up.
- **Footer** (lines 156–167) — brand mark and copyright line.

The interesting bit for students: because `paths` and `features` are just plain arrays that get mapped over, this page is a good example of "data drives the UI" instead of copy-pasting near-identical blocks of markup four or five times.

### `components/layout/MainLayout.tsx`

`MainLayout` — lines 12–71

This is the wrapper component that puts the sidebar (the left-hand navigation menu) next to the actual page content. Every logged-in page (dashboard, upload, roadmaps, quizzes, settings, etc.) renders inside this wrapper so they all get the same navigation shell for free. It takes one prop: `children` (line 12) — this is a standard React pattern where whatever page content is nested inside `<MainLayout>...</MainLayout>` gets rendered in the `{children}` slot (line 67).

Two pieces of state it manages:
- **`collapsed`** (line 13) — whether the desktop sidebar is shrunk down to just icons or shown full-width. This preference is saved to `localStorage` (the browser's small on-device storage that survives page reloads) under the key `"lb-sidebar-collapsed"` (line 10), so if a user collapses the sidebar, it stays collapsed next time they visit. There's a comment (lines 16–18) explaining why the saved preference is read inside a `useEffect` (a hook that runs after the page first renders) instead of during the initial render — reading `localStorage` immediately would make what the server rendered not match what the browser then shows, causing a flicker/mismatch.
- **`mobileOpen`** (line 14) — whether the mobile navigation drawer (a panel that slides in from the side) is open, since on small screens there isn't room for a permanent sidebar.

On desktop, it renders the `Sidebar` component directly (line 34). On mobile, it instead shows a slim top bar with a hamburger menu button (lines 37–53) that opens a `Drawer` (a slide-out panel) containing `SidebarInner` (lines 56–58) — this is the same sidebar content, just displayed differently for a small screen. The actual page content sits in a `<main>` tag (lines 61–68) whose left margin shifts depending on whether the sidebar is collapsed, so the content never sits underneath the sidebar.

### `components/sidebar/Sidebar.tsx`

This file exports two components that work together:

**`SidebarInner`** — lines 40–193 — this is the actual content of the sidebar (the logo, the menu links, the account menu at the bottom). It's pulled out as its own component specifically so it can be reused in two places: the permanent desktop sidebar and the mobile slide-out drawer (see `MainLayout.tsx` above) — one set of code, two places it appears, instead of maintaining the menu twice.

The menu links (defined in the `NAV` array, lines 24–29) are:
- **Dashboard** → `/dashboard`
- **Upload Resume** → `/upload`
- **My Roadmaps** → `/roadmaps`
- **Quizzes** → `/quizzes`

**How it knows which link is "active" (highlighting the current page):** line 47 calls `usePathname()`, a Next.js hook that returns the current page's URL path (e.g. `/dashboard`). Then for each link in the loop (line 98), it checks `pathname === item.href` — if the current URL exactly matches that link's URL, `active` is `true`, and the link gets a different style: a colored gradient background and white text/icon (lines 109–111 and 114) instead of the plain gray "not selected" look. So highlighting isn't some special flag stored anywhere — it's recalculated on every render just by comparing the current URL to each link's URL.

Below the menu is an account section (lines 129–190) built with a dropdown menu component (`DropdownMenu` from Radix, a UI library the app already uses for accessible dropdowns). Clicking the user's avatar/initials opens a menu with:
- **Account Settings** → `/account-settings`
- **Change Password** → `/change-password`
- **Settings** → `/settings`
- **Logout**

**The logout button's behavior** (the `doLogout` function, lines 62–67): when clicked, it (1) calls `clearAuthData()` to wipe the stored access/refresh tokens, (2) calls `logout()` from the shared Zustand auth store to clear the logged-in user's data from app state, (3) redirects the browser to `/login` with `router.push("/login")`, and (4) calls `onNavigate?.()` — an optional callback (only used on mobile) that closes the slide-out drawer after navigating away.

**`Sidebar`** (the default export) — lines 196–220 — this is the actual fixed, permanent sidebar shown on desktop screens (`lg:block` — only visible on large screens and up). It just wraps `SidebarInner` and adds the little collapse/expand arrow button (lines 211–217) that toggles between the full-width and icon-only views.

A small helper worth knowing about: `initials()` (lines 31–37) computes the one or two letters shown in the round avatar badge — it takes the user's first name's first letter, or falls back to the first letter of their email, or falls back to `"U"` if neither exists.

### `components/SkillGap.tsx`

`SkillGapSection` — lines 6–71

This component shows the "Skills You Have" vs. "Skills to Gain" comparison for a roadmap — two side-by-side cards that visually answer "what do I already know, and what am I still missing for this role?"

**Props (inputs) it takes:**
- `extractedSkills` (optional array of strings) — the skills the AI detected from the user's uploaded CV.
- `missingSkills` (optional array of strings) — the skills the target role needs that weren't found in the CV.

Both are optional, so line 13–14 default them to empty arrays if not passed in (`|| []`), avoiding a crash if the data isn't there yet.

**Logic worth pointing out:**
- Line 16: if there are zero skills in both lists, the whole component renders nothing (`return null`) — no point showing two empty cards.
- The left card (lines 21–42) is green-themed and lists skills the user already has, using a `CheckCircle2` (checkmark) icon. If the list is empty, it shows "No skills detected from your CV." instead of an empty box.
- The right card (lines 45–68) is amber/orange-themed and lists the skills still missing, using a lightning bolt (`Zap`) icon. If the gap list is empty, instead of "Skills to Gain (0)" looking broken, it shows a friendly "You're all caught up for this role." message (lines 64–66).
- Each individual skill is rendered as a `Badge` (a small pill-shaped label) — green-toned (`tone="success"`) for skills the user has, amber-toned (`tone="warning"`) for skills they're missing.

This component doesn't fetch any data itself — whatever page uses it (the roadmap view) is responsible for passing in the two skill arrays as props.

### `components/providers/query-provider/index.tsx`

`Providers` — lines 6–24

TanStack Query, in one sentence: it's a library that fetches data from the backend and automatically caches, re-fetches, and keeps that data in sync for you, so you don't have to hand-write the "loading / error / stale data" logic yourself every time a page needs server data.

This file sets up TanStack Query for the whole app. It creates one `QueryClient` (the object that actually manages all the cached data) using `useState`'s lazy-initializer form (`useState(() => new QueryClient(...))`, lines 7–17) — this is a deliberate pattern to make sure the client is only created **once** per app session, not re-created on every re-render.

Two settings are configured on that client (lines 10–15):
- **`staleTime: 60 * 1000`** — data fetched by a query is considered "fresh" for 60 seconds (1 minute). During that window, if another part of the app asks for the same data, TanStack Query just hands back the cached copy instead of hitting the backend again.
- **`refetchOnWindowFocus: false`** — by default, TanStack Query automatically re-fetches data whenever the browser tab regains focus (e.g. the user switches back from another tab). This turns that behavior off, so switching tabs doesn't trigger a burst of extra network requests.

The `Providers` component itself just wraps the rest of the app in `<QueryClientProvider client={queryClient}>` (lines 20–22), which is what makes the query client available to every component in the tree via React Context, without it needing to be passed down as a prop manually.
