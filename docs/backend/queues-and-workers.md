# Backend — Background Job Queue & Worker

## What this is

Turning a CV into a career roadmap is slow: the backend has to call an AI model to generate the roadmap, and then, for every single step in that roadmap, search the web for a matching YouTube video and article. That can easily take many seconds. If the backend tried to do all of that inside the same request that handles the CV upload, the user's browser (and the person sitting in front of it) would just sit there frozen, waiting, possibly until the connection times out. So instead, the backend does the fast part immediately — saving the upload and remembering "there's a job to do" — and replies to the browser right away with something like "got it, I'll let you know when it's ready." The actual slow work happens separately, in the background, in a **queue** (a waiting line of jobs) that a completely different program called the **worker** picks jobs off of, one at a time, and processes at its own pace. This queue is powered by a library called **BullMQ**, which uses **Redis** (a very fast in-memory database — meaning it keeps its data in RAM instead of on disk, so reads and writes are close to instant) to actually keep track of who's waiting in line. The one thing that trips people up constantly: the worker is a **separate process** that must be started manually by running `node workers/cv.worker.js` in a terminal — it does **not** start automatically when you run the normal backend server (`npm run dev`). If you upload a CV and nothing ever happens, the first thing to check is always whether the worker is actually running.

## File by file

### `backend/queues/cv.queue.js`

This file's whole job is to create the queue object that the rest of the backend uses to add new CV-processing jobs to the waiting line. It doesn't do any of the actual CV processing itself — that's the worker's job (next section).

- **Lines 1–5**: imports. `Queue` (line 1) comes from BullMQ and is the class used to create/talk to a named queue. `IORedis` (line 2) is the library that actually opens a connection to Redis. `dotenv` (lines 3, 5) loads secret/config values (like `REDIS_URL`) from a `.env` file into `process.env` so the code below can read them.
- **Lines 7–20**: sets up the connection to Redis. If a `REDIS_URL` environment variable is set (line 9), it connects to that address (this is how you'd point it at a real, possibly cloud-hosted, Redis instance). If not, lines 15–19 fall back to connecting to Redis running on the same machine (`127.0.0.1`, port `6379` — the default Redis port). `maxRetriesPerRequest: null` (lines 11, 18) tells the Redis client to keep retrying forever rather than giving up after a few attempts — reasonable for a queue connection that should stay alive indefinitely.
- **Lines 22–31**: creates the actual queue, named `"cv-processing"` (line 22) — this exact name matters, because the worker (in the next file) has to listen on a queue with the *same* name to receive these jobs; it's how the two separate files/processes agree on which waiting line they're both talking about. `defaultJobOptions` (lines 24–30) sets rules that apply to every job added to this queue:
  - `attempts: 3` (line 25): if a job fails (throws an error), BullMQ doesn't just give up — it automatically tries running it again, up to 3 times total, before finally marking it as failed for good.
  - `backoff: { type: "exponential", delay: 5000 }` (lines 26–29): this controls how long BullMQ waits between those retry attempts. "Exponential backoff" means each retry waits longer than the one before it (roughly: wait 5 seconds, then next retry wait 10 seconds, then 20 seconds, doubling each time), instead of retrying immediately over and over. This gives whatever caused the failure (e.g. the AI service being briefly overloaded) a chance to recover before trying again.
- **Lines 33–36**: registers a listener purely for debugging — it logs `"[Queue] Job added:"` and the job's data to the console whenever this event fires, so a developer watching the terminal can see jobs being queued.
- **Line 38**: exports the finished `cvQueue` object so other files (like the CV upload controller) can call `cvQueue.add(...)` on it to actually push a new job into the waiting line. (Note: this file only *defines* the queue — the code that calls `.add()` with the job's data, such as the uploaded CV's URL and the user's chosen role, lives in the upload controller, not in this file.)

### `backend/workers/cv.worker.js`

This is the separate program mentioned above — the one you have to start manually with `node workers/cv.worker.js`. It's the thing that actually does the slow work: watching the queue, and processing each CV-to-roadmap job that shows up on it.

- **Lines 1–8**: imports. Same idea as the queue file — `dotenv` (lines 1–2) for config, `Worker` from BullMQ (line 3, the class that actually *consumes*/processes jobs, as opposed to `Queue` which only *adds* them), `IORedis` (line 4) for the Redis connection, `axios` (line 5) to make HTTP calls out to the AI service, `mongoose` (line 6) to talk to MongoDB directly, the `Job` model (line 7, the database record that tracks a CV-processing job's status and eventual result), and `extractTextFromCV` (line 8), a helper that runs OCR (Optical Character Recognition — reading text out of an image or scanned document) on the uploaded CV file.
- **Lines 10–25**: connects to Redis — the exact same logic as `cv.queue.js` (env var if set, otherwise local fallback), just with two extra `console.log` lines (13, 19) noting which path was taken. This has to point at the *same* Redis as the queue file, otherwise the worker would never see the jobs the queue is adding.
- **Lines 27–30**: creates a shared `axios` client pre-configured with the AI service's base URL (`FASTAPI_URL` from the environment) and a timeout (`ROADMAP_TIMEOUT_MS`, defaulting to 30 seconds) — used later to call the AI service without repeating this setup every time.
- **Lines 32–128**: the `init()` function — everything that happens when the worker starts up.
  - **Line 34**: first, connects to MongoDB (`mongoose.connect`), since the worker needs to read and write `Job` documents directly (unlike a normal API request, there's no Express request/response cycle handling this connection for it).
  - **Lines 36–114**: creates the actual `Worker` that listens on the `"cv-processing"` queue (line 37 — matching the name used in `cv.queue.js`) and defines the function that runs for every job it picks up (lines 38–112). Walking through that function step by step, in the order it actually runs for one job:
    1. **Line 40**: pulls the data out of the job — `jobId` (which `Job` document in MongoDB this is for), `cvUrl` (where the uploaded CV file lives), `role` (the target career role the user picked, defaulting to `null` if none was given), and `fileType` (defaulting to `"image"`).
    2. **Lines 42–44**: logs the full job data and the extracted `role` value to the console — debug logging left in to help diagnose issues with the `role` field specifically (a comment on line 39 hints this was a known trouble spot).
    3. **Line 46**: a safety check — if there's no `jobId` at all, it immediately throws an error rather than trying to process a job it can't even report status back for.
    4. **Line 49**: marks the job as `"processing"` in the database (updates the `Job` document's `status` field) — this is what lets the frontend, which is polling for job status, show the user "we're working on it."
    5. **Lines 51–58**: gets the CV's text content. For PDF/DOCX files, the text is often already extracted earlier, at upload time, and stored on the job — line 54 checks for that and reuses it if present, to avoid redoing work. Only if there's no text yet (e.g. the file is an image, or a scanned PDF with no selectable text layer) does it run the actual (slow) OCR step via `extractTextFromCV` (line 56), then save that extracted text back onto the `Job` document (line 57) so it isn't lost.
    6. **Lines 60–65**: builds the payload to send to the AI service — the `jobId`, the extracted CV text, and the `role`.
    7. **Lines 67–72**: more debug logging — prints the exact payload being sent, its keys, and specifically double-checks whether `role` ended up `undefined` or `null`. Again, this looks like leftover debugging around a `role`-related bug rather than something needed for normal operation.
    8. **Lines 74–89**: sends the payload to the AI service (`POST /ai/roadmap`, using the shared `axiosClient` from earlier) and waits for the response. If that call succeeds, line 81 logs the response data. If it fails (network error, AI service error, etc.), lines 82–88 log the error message and, if the AI service sent back an error response, its status code and body too — then re-throws the error so it's caught by the outer `catch` block (see step 10 below).
    9. **Lines 91–98**: once the AI service responds successfully, saves the results onto the `Job` document — the generated `roadmap`, `tags`, `extractedSkills`, `requiredSkills`, and `missingSkills` (each falling back to `null`/an empty array if the AI service didn't return that field) — and sets `status` to `"completed"`. This is the finish line: once this write happens, the frontend's polling will see the job is done and can display the roadmap.
    10. **Lines 99–110**: the `catch` block for the whole `try` above (started on line 48). If anything went wrong at any step — OCR failure, the AI service call failing, etc. — this updates the `Job` document's `status` to `"failed"` and stores the error's message on the `error` field (lines 101–105), so the frontend can show the user what happened instead of leaving them stuck on "processing" forever. If *that* update itself somehow fails, line 108 just logs it rather than crashing. Line 110 then re-throws the original error — this is important because BullMQ needs to see the job actually throw in order to know it failed and trigger a retry (the `attempts`/`backoff` settings from `cv.queue.js`).
  - **Line 113**: passes the Redis `connection` into the `Worker`, so it listens on the same Redis instance the jobs are being added to.
  - **Lines 116–127**: sets up a clean shutdown — if the process receives `SIGINT` or `SIGTERM` (the signals sent when you press Ctrl+C or when something asks the process to stop, lines 126–127), it disconnects from Redis and MongoDB (lines 118–119) before exiting, rather than just being killed mid-operation.
- **Lines 130–133**: actually calls `init()` to kick everything off. If `init()` itself fails (e.g. can't connect to MongoDB at all), it logs the error and exits the process (`process.exit(1)`) rather than limping along in a broken state.

**Debug logging left in the code**: this file has quite a bit of `console.log`/`console.error` calls that look like leftover debugging rather than intentional, permanent logging — lines 13/19 (which Redis connection path was used), 42–44 and 67–72 (dumping the full job data and payload, specifically interrogating the `role` field), 81 (the AI response), and 83–87 (AI call error details). None of these are harmful, but they're a lot of console noise for normal operation, and they're a good example of why the root project notes call out that this codebase has no structured logging library yet — see `backend/CLAUDE.md`.
